ReceiptKit — a cloud-to-thermal-printer platform I architected and built end to end.

Where the problem came from:

I architected a custom POS and business-management system for a two-location hardware retailer. They use Star Micronics thermal printers, and the owner wanted a cloud-based receipt printing solution that supported images and custom fonts. Cloud printing is doable, but the genuinely hard part is doing it without noticeable print delays while supporting image and font data within the receipt. Solving that properly eventually became its own product.

I worked through the options that existed at each point in time, and eventually settled on the ReceiptKit solution:

1. Star's CloudPRNT — the available approach for cloud printing at the time. The printer polls a cloud endpoint every ~5 seconds for queued jobs, so I built a cloud print server around it. It worked, but prints landed 7–15 seconds after checkout depending on polling timing and render cost. Employees complained constantly.

2. Cloud edge functions + a queue DB. I moved the print server to serverless and optimized the image-to-binary conversion. This got us to ~7–10 seconds and removed a standalone VM — but the 5-second polling meant a serverless invocation every 5 seconds per printer, roughly $5–10/printer/month. Tolerable for one store, but a non-starter as a SaaS: the cost scales with idle printers, not usage.

3. Star's newer CloudPRNT Next (on-printer MQTT), once it was released. It required buying new hardware. When I tested it, text-only receipts were fast, but anything with an image — a logo, a barcode, a QR code — took 5+ seconds, and got worse with more image data. I worked directly with Star Micronics to identify the bottleneck, and we eventually concluded that the printer hardware simply can't process image data fast enough over the MQTT route — the delay was something we'd have to accept.

The decision that became ReceiptKit:

I then sent the same image receipt data directly to the printer over TCP and USB, which started printing instantly as soon as the data started streaming in. This formulated a workable SaaS architecture: a small local bridge on the store's network that speaks TCP/USB to the printer, speaks MQTT to the cloud, and does image rendering locally.

The bridge also caches the receipt template, and that single decision solves both problems that had made every prior approach non-viable. Because the template lives on the bridge, a print request only sends the small JSON payload needed to populate it — the dynamic data — instead of a large rendered image receipt for every print. That's a huge bandwidth reduction. And because the rendering and format conversion happen on the bridge, there's no per-print edge invocation and no server compute cost — all of it is offloaded to a device the customer already has on their network. That's what makes ReceiptKit both fast and economically viable as a SaaS, which no cloud-hosted approach could be.

The main technical tradeoff — the render format:

A SaaS needs dynamic receipts, so I designed an "SVG pseudo-template": static SVG plus element definitions, with a fast text transform that injects the JSON data to produce a real SVG, which then rasterizes to the printer's binary format. I chose SVG because browsers render it natively (great for the live editor) and it converts to other formats extremely cheaply. The big win was fonts: instead of a rasterizer library that loads system fonts — which cost ~600ms per render — I do the font-to-outline conversion myself, which cut render time to ~12ms, about 50x.

A second tradeoff — Rust vs. one codebase:

For the Raspberry Pi bridge I rewrote the hot rendering path as a native Rust binary for maximum speed; the desktop bridge instead reuses the exact JavaScript generator the website uses (embedded via QuickJS). The tradeoff was deliberate: I accepted maintaining a performance-critical path in a second language, in exchange for the fastest possible render on the always-on device, while keeping a single source of truth for the common case.

Impact:

End-to-end print time went from 7–15 seconds to under 100ms (~40ms on the local network) — roughly a 100x improvement over the production system it replaced — with full support for images and custom fonts and no per-print polling or compute cost. It solved a problem that the printer vendor's own two cloud products couldn't, and it's now the foundation of ReceiptKit, an early-stage commercial launch.
